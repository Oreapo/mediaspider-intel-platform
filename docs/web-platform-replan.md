# Web Platform Replan

## 新定位

该 Web 平台的目标不是做一个“抓取控制台”，而是做一个**反黑灰情报工作台**。

平台的主工作流应当是：

1. 创建采集任务
2. 获取线索数据
3. 沉淀为数据集
4. 抽取风险信号
5. 聚合成风险实体
6. 归档到案件
7. 导出证据包

## 设计原则

### 原则一：先线索，后分析

分析是为风险识别服务，不是单独的 BI 模块。

### 原则二：先对象，后报表

平台的核心对象不是“图表”，而是：

- 任务
- 数据集
- 信号
- 实体
- 案件
- 证据

### 原则三：场景优先

任务和页面都应该围绕反黑灰场景组织，例如：

- 引流导流
- 灰产招募
- 商品风险
- 卖家风险
- 话题监测

### 原则四：案件化闭环

最终成果不只是页面上的结果卡片，而是能够进入案件和证据包。

## 页面重排建议

### 第一层：工作入口

- `Dashboard`
- `Tasks`
- `Datasets`

### 第二层：风险研判

- `Signals`
- `Entities`
- `Analysis`

### 第三层：案件处置

- `Cases`
- `Evidence`

### 第四层：支撑页

- `Logs`
- `Settings`

## 每个页面的核心问题

### Dashboard

今天新增了什么高风险线索？

### Tasks

下一批线索要从哪里抓、按什么规则抓？

### Datasets

这批数据是否足以支持后续风险识别？

### Signals

哪些记录已经命中可疑规则？

### Entities

这些信号背后是不是同一批账号 / 卖家 / 商品 / 联系方式？

### Analysis

从平台专属视角和跨平台视角能得出什么研判？

### Cases

这些信号和实体如何组织成一个可持续追踪的案件？

### Evidence

最终有哪些可导出的证据、截图、原始记录与摘要？

## 实施顺序

### Step 1

在现有骨架上增加：

- `signals`
- `entities`
- `cases`

### Step 2

把现有 `tasks / datasets / analysis` 页面改成为案件工作流服务。

### Step 3

增加证据包与告警能力。

### Step 4

再做平台专属分析器和关系图谱增强。

## 当前结论

后续代码开发优先顺序应当调整为：

1. `signals`
2. `entities`
3. `cases`
4. `evidence`
5. 再扩充图表和复杂可视化
